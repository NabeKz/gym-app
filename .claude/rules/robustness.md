# ロバストネス図の書き方

## 各要素の役割

| 要素 | 役割 | 名前の形式 |
|------|------|-----------|
| boundary | アクターが触るインターフェース | 「〜する」（例: 予約をキャンセルする） |
| control | ビジネスロジック・条件チェック | 「〜する」（例: 予約の存在と所有権を確認する） |
| entity | データオブジェクト | 名詞のみ（例: Reservation） |

## 書かないこと

- **HTTP エンドポイントパス** — `DELETE /reservations/{id}` は OpenAPI の領域
- **カラム詳細** — `(id, member_id, lesson_id)` は ER 図の領域
- **エラーレスポンス** — 401 / 403 / 404 などはシーケンス図の領域
- **認証・認可** — ミドルウェア層の関心事であり、フィーチャーのロバストネス図に含めない

## control の省略

boundary と control が同じ意味になるなら control は不要。

```
# 冗長（避ける）
boundary "予約をキャンセルする" --> control "キャンセルを実行する"

# シンプルで十分な場合は boundary → entity を直結する
```

## 詳細なケースはテストで管理

条件分岐ごとのエラーケースは robustness 図に書かず、`todo` 付きテストとして管理する。

```gleam
pub fn cancel_not_own_reservation_test() {
  todo
  // 他会員の予約 → 403
}
```

## 認証が必要なユースケース

認証済み前提で図を書く。未認証ケースはミドルウェアのテストに委ねる。
